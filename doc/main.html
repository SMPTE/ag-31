Document repository

Usage

A GitHub repository is primarily intended for:

  • tracking issues against engineering documents and publishing Public Committee Drafts (PCD);
  • managing engineering documents via the HTML Workflow as defined by ;
  • managing non-prose element(s) of engineering documents that are intended to be made publicly available, including source code, XML, JSON, etc.

Private vs public

A repository can either be:

  • public; in which case it is accessible by the public
  • private; in which case it is accessible only by SMPTE Standards Community Members

Unless requested by the project group, repositories are created as public repository.

A public repository should remain public.

Discussions in public repository are not subject to the SMPTE confidentiality policy.

A project group may request that a repository be created as private, or made private.

A repository that is private shall not be made public, unless explicitly approved by the Standards Vice-President as provided by the SMPTE confidentiality policy.

In the event that a public repository is created when a private repository already exists, the private repository should be archived and -private appended to its name.

Creation

A repository is created only on request to sent to githubsteering@smpte.org.

Repo Types

Main Document Repo

General

There is one public repository for all editions and stages of a document. This repo is primarily used for collecting issues from the public.

The main benefit of this repo is that anyone in the community may make a comment (as an issue) about a document without needing to be a SMPTE participant, as long as they are a GitHub user.

This repo may change via a PR for publication of a PCD, and back to a basic issue tracking repo via a PR after completion of a PCD. For guidance on this, see

Naming

The syntax for repo naming is:

("rp" | "st" | "eg")<document root number> "-" <document part>

The repository for the document "ST 2067-105" is named st2067-105

HTML Publishing Document Repo

General

There is zero or one private HTML document repository for all editions and stages of a document, created from the template at as defined by .

This repo is used to maintain the HTML source of a document, if one exists, and any additional elements that do not require their own repositories. The workflow for this document process is managed and documented at .

Issues may not be collected in this repo, and the GitHub issue tracker is disabled as such. Editors should always be aware of issues in the Main Document Repo, which may come from the public.

Currently, these repos will always remain private.

Naming

The syntax for repo naming is:

("rp" | "st" | "eg")<document root number> "-" <document part> "-private"

The repository for the HTML workflow of the document "ST 2067-105" is named st2067-105-private

User access

GitHub Teams

Users shall be set into varios GitHub Teams based on need, such as:

  • One team per TC
  • One team for ST
  • One team per TC chair team

GitHub Permissions

Users and Teams shall have the permissions:

  • Parent TC team -> read
  • Parent TC chairs team -> admin
  • Project group chairs/editors (as contributors) -> maintain

GitHub workflow

Project groups should adhere to the following rules:

  • one or more editors are responsible for making modifications to the repo;
  • every substantial modification should be done through a pull request;
  • a pull request should require at least one approval from a project group member before being merged;
  • a pull request should only be merged with the consensus of the project group, which can be obtained using the following methods:
    1. offering the pull request for a review period and the review period resulting in no substantial objections (a review period of 2 weeks is recommended);
    2. seeking explicit project group consensus, e.g. during a meeting or via email;
    3. granting the editor(s) to make modifications without explicit project group consensus, e.g. early in the development of the document.
